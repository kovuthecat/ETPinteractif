## T1-E — Temps ① « Couvrir le repas » : le résultat dépend de (dose − glucides), pas de la dose seule

### Objectif
Corriger l'incohérence de fond (point 11) : aujourd'hui « Dose habituelle » couvre **toujours** parfaitement le
repas quelle que soit la charge. Cible : « Dose habituelle » = dose **fixe** calibrée pour un **repas moyen** ; le
résultat suit l'écart (dose − glucides).

### Décision clé
Dans `t1Points.avec`, la dose vaut `cran.params.charge * DOSE_FACTOR[doseCouvrir]` → elle se recale sur la charge
du repas, d'où le bug. La rendre **fixe** en la calant sur le repas moyen, **exactement comme le temps ③** le fait
déjà : `dose: DOSE_ADEQUATE * DOSE_FACTOR[doseCouvrir]` (indépendante de `cran.params.charge`). Effets attendus
(le repas garde son amplitude propre via `cran.params.charge`, la dose non) :

| | Moins | Habituelle (fixe = moyen) | Plus |
| --- | --- | --- | --- |
| **Peu de glucides** | ~cible | **HYPO** (dose > repas) | hypo marquée |
| **Repas moyen** | reste haut | **cible (seul cas parfait)** | hypo |
| **Beaucoup de glucides** | très haut | **reste haut / hyper** | ~cible |

Puis `messageCouvrir` devient fonction **des deux** axes : `messageCouvrir(repasId, doseCouvrir)` — une phrase par
case cohérente avec la courbe (`// à revalider (Thibault)`). **Priorités** (rapport) : d'abord « Peu + Habituelle »
(→ hypo) et « Beaucoup + Habituelle » (→ reste haut, **ne doit pas plonger**).

### Lire / Modifier
- Modifier : `InsulineRapideModule.tsx` (formule de dose de `t1Points.avec` ; signature + corps de
  `messageCouvrir`). Éventuellement `DOSE_FACTOR`/`DOSE_ADEQUATE` si le calage visuel l'exige (constantes du module).

### Étapes
1. Remplacer la dose de `t1Points.avec` par `DOSE_ADEQUATE * DOSE_FACTOR[doseCouvrir]` (fixe, calibrée moyen).
2. Réécrire `messageCouvrir` en matrice `(repasId, doseCouvrir)` (9 cases), en priorisant les 2 cases citées.
3. Vérifier visuellement (validation Thibault) que Peu+Habituelle plonge (hypo) et Beaucoup+Habituelle reste haut ;
   ajuster `DOSE_ADEQUATE`/`DOSE_FACTOR` **au module** si le contraste n'est pas net (jamais de chiffre à l'écran).

### Validation
- Auto (bloque le commit) : `npx tsc --noEmit` · `npm run build` · `npm test` → verts.
- Humain (visuel) : à « Dose habituelle », Peu → hypo, Moyen → cible, Beaucoup → reste haut ; message cohérent avec
  la courbe dans chaque case. → `VALIDATION.md`.

### Si bloqué
Si le contraste hypo/haut n'est pas atteignable sans afficher de chiffre ni casser les autres temps → STOP,
décrire le compromis.

## T2-E — Temps ③ « Corriger avant le repas » : réglages fins

### Objectif
Trois recalages (point 12) sur un onglet dont la logique est déjà saine (dépend bien des deux axes).

### Décision clé
1. **« Dans la cible + Dose habituelle »** : le creux passe très légèrement sous la cible (≈ limite hypo) → recaler
   dans le vert. Levier : `DOSE_ADEQUATE` (léger −) ou `BOLUS_EFFET_MAX` (lib), à caler à l'œil.
2. **« Basse + Moins de dose »** : reste franchement en hypo malgré dose réduite + apport du repas → un léger retour
   vers la cible est plus réaliste (le message « traiter l'hypo d'abord » reste juste). Levier : `DEPART_RESORPTION`
   / valeur `basse` de `DEPART_OPTIONS`.
3. **Départ « Haute »** posé pile sur la limite cible/hyper → le démarrer **nettement** dans le rouge : monter la
   `value` de l'option `haute` dans `DEPART_OPTIONS` (aujourd'hui `BASELINE + 30`).

- Éventuellement : `glycemieCurve.ts` (`BOLUS_EFFET_MAX`, `DEPART_RESORPTION`) si le calage l'exige — **alors
  vérifier/ajuster** `glycemieCurve.test.ts`.

## T3-E — Temps ④ situation « redescend toute seule » : redessiner la courbe de base

### Objectif
Dans la situation « La glycémie redescend toute seule », la courbe de base « je n'ajoute pas de dose » reste quasi
plate dans la cible → elle ne montre pas la dynamique attendue (point 13). Attendu : **départ cible → montée nette
post-prandiale (zone haute) → redescente spontanée dans la cible vers +3h**, sans dose ajoutée.

### Décision clé
Aujourd'hui `t4Points.sansRecorrection` pour la situation `revient` = `sampleRepasAvecBolus(REPAS_MOYEN,
{dose: DOSE_ADEQUATE, tInjection})` : une dose adéquate **aplatit** le repas → pas de montée visible. Pour la
situation `revient`, la base doit montrer l'**excursion post-prandiale naturelle** qui revient seule : réduire (ou
retirer) le bolus de la base `revient` de sorte que le repas monte dans le rouge puis retombe vers la cible via le
retour naturel du repas (`repasLevelAt` revient à baseline entre `endTimeMinutes` ~120-180 min → cadre bien avec
« +3h »). Garder les options de recorrection **inchangées** (elles plongent : ajout de dose non nécessaire → hypo,
cohérent). Le message `matriceCumul('revient','aucune')` (« la dose de départ suffisait ») reste valable si l'on
garde une petite couverture ; sinon reformuler légèrement (`// à revalider`).

### Lire / Modifier
- Modifier : `InsulineRapideModule.tsx` (`t4BolusBase` / la dose de la base pour `situationCumul === 'revient'` ;
  ajuster le message `matriceCumul` de la case `revient/aucune` si besoin).

### Étapes
1. Pour la situation `revient`, réduire/retirer la dose de la courbe **de base** (« aucune » recorrection) afin de
   faire apparaître la montée haute puis le retour spontané en cible vers +3h.
2. Vérifier que les deux options de recorrection continuent de plonger en hypo (inchangé).
3. Ajuster le libellé de la case `revient/aucune` si la nouvelle courbe le demande (`// à revalider`).
