## T9 — Point 11 : courbes de départ **convergentes** + correction expérimentable (onglet ③)

### Objectif

Corriger la cause du point 11 : `depart` est aujourd'hui un **décalage vertical constant** → les 3
courbes (Basse/Cible/Haute) ont la **même forme, translatée**. Faire que l'écart de départ **se résorbe**
(forme convergente) ; puis, sur l'onglet ③, rendre la **correction expérimentable** (ajouter de la rapide
sur une glycémie haute → voir la courbe redescendre dans la cible).

### Décision clé

Dans `sampleRepasAvecBolus` (l.318-331), `repas = repasLevelAt(t) + decalageDepart` applique l'écart à
**tout** `t`. Le remplacer par un écart qui vaut plein avant le repas puis **décroît (ease) vers 0** →
une glycémie haute revient vers la baseline (forme distincte, convergente). Ce changement **garde tous
les tests existants verts** (l'écart à `t=-20` reste plein ; le pic reste plus haut) ; on **ajoute** des
invariants de convergence.

### Lire / Modifier

- Modifier : `glycemieCurve.ts` (`sampleRepasAvecBolus` uniquement + 1 constante), `glycemieCurve.test.ts`
  (ajouts), `InsulineRapideModule.tsx` (onglet ③), `InsulineRapideModule.module.css`

### Étapes

1. **`glycemieCurve.ts`** — ajouter une constante près des autres constantes bolus (l.287-297) :
   ```ts
   /** Résorption de l'écart de glycémie de départ (min) : un départ haut/bas revient vers la
    *  baseline, plutôt qu'un décalage constant → formes convergentes. `// à caler (Thibault)`. */
   const DEPART_RESORPTION = 180;
   ```
   Puis, dans `sampleRepasAvecBolus`, remplacer le calcul de `repas` :
   ```ts
   const decalageDepart = (bolus.depart ?? BASELINE) - BASELINE;
   // …
   return sampleRange(-20, tEnd, 1, (t) => {
     const deviation =
       t <= 0 ? decalageDepart : decalageDepart * (1 - ease(clamp01(t / DEPART_RESORPTION)));
     const repas = repasLevelAt(params, t) + deviation;
     let effet = bolusEffet(t - bolus.tInjection, bolus.dose);
     if (bolus.tSecondeDose !== undefined) effet += bolusEffet(t - bolus.tSecondeDose, bolus.dose);
     return clampRange(repas - effet, 0, LEVEL_MAX);
   });
   ```
2. **Tests** — dans `describe('invariant 10 …')`, **ajouter** (sans toucher aux tests existants, qui
   restent verts) :
   - **Convergence** : `depart = BASELINE + 20` et `depart = BASELINE` (même dose/timing) → l'écart
     `|valueAt(haut, t) − valueAt(defaut, t)|` à `t ≈ 170` est **nettement plus petit** qu'à `t = -20`
     (l'écart s'est résorbé).
   - **Forme distincte** : les deux courbes ne sont **pas** une translation constante → l'écart à
     `t=-20` diffère de l'écart à `t≈120` (au-delà d'une tolérance).
3. **Composant, onglet ③** (l.294-331) : garder les 3 chips départ (Basse/Cible/Haute). Ajouter un état
   `const [corriger, setCorriger] = useState(false)` (réinitialisé quand `departId` change).
   - `t3Points.selection` (l.164-168) : `dose = DOSE_ADEQUATE + (departId === 'haute' && corriger ? CORRECTION_DOSE : 0)`,
     avec `const CORRECTION_DOSE = 0.4; // à caler (Thibault)`. `depart = departValue` inchangé.
   - Quand `departId === 'haute'` : afficher un contrôle d'expérimentation « Ajouter une correction de
     rapide » (toggle, `aria-pressed`, `activeDoubled`) → au clic, la courbe haute **redescend dans la
     bande cible** (impact visible). Message adapté (l.328-330) : sans correction « ça reste haut » ;
     avec correction « un peu plus de rapide ramène vers la cible ».
   - `departId === 'basse'` : **inchangé** — pas de correction, porte « Traiter l'hypo d'abord »
     (`onNavigate('hypoglycemie')`, l.319-326). `departId === 'cible'` : message inchangé.
4. **CSS** : styliser le toggle de correction (cible ≥ 44 px, `activeDoubled`).

## T10 — Point 12 : le cumul, deux situations cliniques expérimentables (onglet ④)

### Objectif

Audit §12, **décision de contenu Thibault (2026-07-12)** : l'onglet ④ oppose **deux situations** qu'on
peut expérimenter, au lieu d'un toggle binaire :
- **A — « La glycémie redescend toute seule »** : si on attend, elle revient dans la cible ; **ajouter
  une dose avant qu'elle soit redescendue → hypo** (le vrai piège du cumul).
- **B — « La glycémie reste haute »** : après le délai, elle est toujours haute → une dose
  supplémentaire **se justifie** ; mais donnée **trop tôt** (1ʳᵉ dose encore active) elle plonge quand
  même. Donnée **après que la 1ʳᵉ a fini d'agir**, elle ramène dans la cible.

### Décision clé — matrice des issues (le cœur pédagogique)

| Situation | Je n'ajoute pas de dose | Je recorrige tout de suite | J'attends que la 1ʳᵉ ait fini, puis je recorrige |
| --- | --- | --- | --- |
| **A** redescend seule | ✓ revient dans la cible | ✗ plonge en hypo (dose inutile + trop tôt) | ✗ plonge en hypo (la dose n'était pas nécessaire) |
| **B** reste haute | reste haute (non corrigée) | ✗ plonge (empilement avec la 1ʳᵉ encore active) | ✓ ramenée dans la cible |

Le modèle supporte **déjà** `tSecondeDose` (2ᵉ dose) → **aucun changement de `glycemieCurve.ts`**. Les
deux situations se distinguent par la **couverture du repas de base** :
- A = repas **bien couvert** (dose ≈ charge) → la courbe revient seule dans la cible.
- B = **gros repas sous-couvert** (charge élevée, 1ʳᵉ dose insuffisante) → la courbe **reste au-dessus
  de la bande cible** dans la fenêtre post-repas.
La recorrection est une `tSecondeDose` (tôt vs après attente), dose de correction modeste.

- Modifier : `InsulineRapideModule.tsx` (onglet ④, l.333-364 ; état `cumulActive` l.117 ; constantes
  `T_SECONDE_DOSE` l.103, `REPAS_MOYEN`/`REPAS_CRANS` l.75-81), `InsulineRapideModule.module.css`

   Réinitialiser `recorrection` à `'aucune'` quand `situationCumul` change.

   - `'reste-haut'` → repas plus chargé **sous-couvert**, ex.
     `sampleRepasAvecBolus({ charge: 0.8, frein: 0.35, retard: 0.3 }, { dose: DOSE_ADEQUATE, tInjection: T_INJECTION_DEFAUT })`.
     *(Valeurs `// à caler (Thibault)` — objectif : la courbe reste **au-dessus de la bande cible**
     jusqu'à l'instant « attente ».)*
3. 2ᵉ dose (recorrection) via `tSecondeDose` (minutes internes, **non affichées**) :

4. Deux contrôles (aucun chiffre, `aria-pressed`, `activeDoubled`, cibles ≥ 44 px) :

5. Courbe + légende recalculées selon (situation, recorrection). La 2ᵉ courbe (variante `fantome`) et le

6. Message adapté à la **case de la matrice** courante (issue ✓/✗ + raison courte), sans chiffre. Porte

### Ce qui a été vérifié

Recherche exhaustive sur grille : `dose` ∈ [0.3 .. 1.5], `PLATEAU_RESORPTION` ∈ [200 .. 1000],
`plateauHaut` ∈ [10 .. 60] (> 500 combinaisons), en exigeant :

- **B + rien** (`aucune`) reste au-dessus de la bande cible (> 60) à l'instant « attente »
  (t=150, repère repas) et ne plonge jamais sous 20 sans correction ;
- **B + tôt** (2ᵉ dose à t=15) plonge nettement sous la bande cible (< 24) ;
- **B + attente** (2ᵉ dose à t=150) atterrit **dans** la bande cible [24, 60] sans la dépasser ni
  plonger dessous.

**Aucune des > 500 combinaisons ne satisfait les 3 conditions simultanément** (0 résultat). Pire :
sur les 405 configurations qui satisfont déjà la 1ʳᵉ condition (« B reste haute »), la case
`attente` est **aussi mauvaise ou pire** que la case `tôt` dans 81 % des cas (329/405) — c'est
l'**inverse exact** de ce que la matrice demande (« tôt » doit être dangereux, « attente » doit
être sûr).

### Cause structurelle (pas un problème de calage)

À l'instant « attente » (t≈150), la courbe `repasLevelAt` du repas B est déjà quasiment revenue à
la baseline (cf. tentative 1) — le seul « coussin » qui reste au-dessus de la baseline à cet
instant, c'est le plateau lui-même. À l'instant « tôt » (t≈15-100, proche du pic du repas), le
coussin est **plus grand** : pic du repas *plus* plateau quasi intact tous les deux. Une dose de
correction unique, assez forte pour transpercer le **grand** coussin de la fenêtre « tôt » (et
donc y provoquer une hypo), transperce nécessairement plus violemment le coussin **plus petit** de
la fenêtre « attente » — provoquant une hypo là aussi, au lieu d'un atterrissage en douceur dans la
cible. À l'inverse, une dose assez douce pour atterrir proprement dans la cible à l'instant
« attente » est trop faible pour faire plonger la courbe à l'instant « tôt ». Extrait de la grille
(REPAS `{charge:0.8, frein:0.35, retard:0.3}`, valeurs de la courbe en niveau 0-100, cible=[25,60]) :

| dose | `PLATEAU_RESORPTION` | `plateauHaut` | aucune (t=150) | tôt (min après injection) | attente (min après injection) |
| --- | --- | --- | --- | --- | --- |
| 0.5 | 600 | 40 | 63,1 (haute, ✓) | 43,2 (ne plonge pas, ✗) | 54,6 (semble ok…) |
| 0.7 | 500 | 40 | 60,3 (haute, ✓) | 27,1 (limite, ~✗) | **15,9 (plonge aussi !)** |
| 0.9 | 600 | 40 | 62,3 (haute, ✓) | 10,4 (plonge, ✓) | **9,3 (plonge presque autant)** |
| 1.0 | 600 | 40 | 62,1 (haute, ✓) | 0,9 (plonge fort, ✓) | **3,9 (plonge quasi pareil)** |

Augmenter la dose fait bien plonger « tôt », mais fait plonger « attente » **tout autant** (les
deux colonnes de droite bougent quasi ensemble) — aucun réglage ne les dissocie dans le sens voulu.

### Ce qui n'a pas été retenu comme solution

- Découpler la dose de la 1ʳᵉ injection de celle de la recorrection (`doseCorrection` séparé) :
  n'aurait pas résolu le problème, puisque la recorrection utilise de toute façon **une seule et
  même valeur** pour les cases « tôt » et « attente » — la cause (coussin plus petit à l'instant
  « attente » qu'à l'instant « tôt ») reste identique quelle que soit la dose choisie.
- Fudger les seuils de lecture de la matrice (assouplir « plonge »/« cible ») : écarté
  explicitement par le mandat (« n'implémente pas une version dégradée silencieusement »).

### Proposition pour arbitrage (non implémentée)

1. **Piste 2** (déjà évoquée tentative 1) : une élévation qui ne se résorbe **que** sous l'effet
   d'une dose de correction réellement administrée, jamais par le seul écoulement du temps. Elle
   lève la cause structurelle ci-dessus : à l'instant « attente », si rien n'a encore été injecté,
   le coussin serait resté **intact** (aussi grand qu'à l'instant « tôt », pas rabougri par un
   décompte temporel) — une dose « attente » y trouverait donc le **même** coussin qu'une dose
   « tôt », mais sans l'effet résiduel de la 1ʳᵉ dose encore active pour l'écraser davantage. C'est
   un changement plus structurant qu'un ajustement de constante (état supplémentaire, pas un
   simple terme additif en fonction de `t`).
2. Revoir la lecture pédagogique de la case « attente » : accepter qu'une recorrection tardive
   reste risquée elle aussi (juste **moins** que « tôt »), et ajuster les libellés/messages en
   conséquence plutôt que d'exiger un atterrissage net dans la cible — décision de contenu, pas de
   modèle.
