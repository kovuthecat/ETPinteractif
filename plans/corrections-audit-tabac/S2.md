## T4 — Grille d'accueil plus attractive + correction a11y

### Objectif

La grille des 21 affirmations est monotone (texte nu, pas de hiérarchie, aucun feedback vu/non-vu, rien n'indique
le clic) et porte un **bug d'accessibilité**. La rendre plus lisible/engageante et corriger l'a11y.

### Décision clé

- **a11y (à corriger) :** `role="listitem"` est posé sur un `<button class="tile">` — incorrect (un `listitem`
  doit être enfant d'un `role="list"`, et ça masque le rôle bouton natif). Corriger : conteneur `role="list"`
  (ou `<ul>`) + enfants `<li>`/`role="listitem"` **enveloppant** le `<button>`, ou retirer le rôle du bouton.
- **UI :** s'inspirer des écrans détail/révélation réussis (mêmes tokens). Ne pas tout réinventer.

### Lire / Modifier

- Modifier : `IdeesRecuesModule.tsx` (grille), `.module.css` (`.grid`, `.tile`, `.tileText`, `.progressText`).

### Étapes

1. **a11y** : corriger la sémantique liste/bouton (voir Décision clé).
2. **Hover/affordance** : effet de survol clair (bordure/ombre) signalant le clic ; coins arrondis, hauteurs de
   carte homogènes, `gap` plus généreux.
3. **Feedback d'état** : marquer les cartes déjà vues (coche / opacité réduite / bandeau « vu ✓ ») ; rendre le
   compteur `.progressText` visuel (barre de progression). L'état « vu » vit dans le state local du module (aucune
   persistance).
4. Optionnel (si simple) : petite icône/illustration en tête de carte (l'`IllustrationSlot` existe déjà).

### Validation

- Auto : `tsc` · `build` · `test` verts.
- Humain : grille aérée, hover évident, cartes vues distinguées, compteur visuel ; lecteur d'écran annonce bien
  une liste de boutons.
- auto OK (tsc/build/test verts) - en attente de validation visuelle par Thibault (npm run dev)

### Si bloqué

Si le feedback « vu » impose de remonter l'état hors du module (partage inter-modules) → **ne pas** le faire ici
(c'est le rôle de S10) ; se limiter au state local de la grille. STOP si ambigu.
