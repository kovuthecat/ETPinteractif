## T11 — `SelectionContext` en mémoire, câblé aux modules et à « Mon plan d'arrêt »

### Objectif

Aujourd'hui les sélections (situations, forme de substitut, outils « Dans ma fiche », raisons) vivent dans le
state **local** de chaque module et sont **perdues** en quittant le module → « Mon plan d'arrêt » repart de zéro.
Introduire un état partagé **en mémoire** qui survit à la navigation inter-modules, et câbler chaque module en
écriture + le plan en lecture.

### Décision clé

La navigation se fait par **état dans `App.tsx`** (pas de router, pas de remontage global) : un **Context monté
au-dessus du switcher de vues** survit au changement de module alors que le state local des modules est détruit.
C'est le mécanisme conforme à « zéro persistance » (pas de `localStorage`).

Forme suggérée de l'état (par thème) :
```ts
interface SelectionState {
  situations: string[];      // ids de situations cochées (Composantes)
  substituts: string[];      // forme(s) retenue(s)
  outilsFiche: string[];     // outils marqués « Dans ma fiche »
  raisons: string[];         // raisons d'arrêter
  gestesEcart: string[];     // « Si j'ai un écart »
  dateArret?: string;
}
```

### Lire / Modifier

- Créer : `src/state/SelectionContext.tsx` (Provider + hook `useSelection()`), **générique** (Map par `themeId`).
- Modifier : `App.tsx` (monter le Provider au-dessus de la vue module) ; les 5 modules ci-dessus (écriture/lecture).

### Étapes

1. Créer le Context + Provider (état en `useState`/`useReducer`, aucune persistance) + hook `useSelection(themeId)`
   exposant lecture + actions (`toggleSituation`, `setSubstituts`, `toggleOutilFiche`, `toggleRaison`, `reset`, …).
2. Monter le Provider dans `App.tsx` au-dessus du rendu de module (il ne doit **pas** se démonter au changement de module).
3. **Écriture** : brancher les toggles/chips existants (situations `AddictionModule`, forme `SubstitutsModule`,
   cases « Dans ma fiche » `BoiteAOutilsModule` — cf. S3 T8, raisons `MotivationModule`) pour écrire dans le Context.
4. **Lecture** : `PlanArretModule` pré-coche/affiche chaque section depuis le Context ; l'édition sur place met à
   jour le Context (cohérence bidirectionnelle).
5. Ajouter une action **« Réinitialiser mon plan »** explicite (pas de remise à zéro implicite à la navigation).
6. Généricité : la clé d'état est le `themeId` courant ; aucun libellé de thème en dur dans le Context.

### Validation

- Auto (bloque le commit) : `npx tsc --noEmit` · `npm run build` · `npm test` verts. **Vérifier l'absence de
  `localStorage`/`sessionStorage`/`document.cookie`** dans le diff (grep).
- auto OK (tsc/build/test verts, grep persistance clean) — en attente de validation visuelle par Thibault
  (`npm run dev`) : cocher des situations (Composantes) / une forme (Substituts → « Retenir pour mon plan ») /
  des outils (« Dans ma fiche ») / des raisons (Motivation) dans leurs modules, puis ouvrir « Mon plan d'arrêt »
  → tout est **pré-rempli** ; revenir/aller entre modules conserve la sélection ; un **rechargement (F5) remet à
  zéro** (attendu) ; « Réinitialiser mon plan » vide l'état.

### Si bloqué

Si un module écrit sa sélection via une structure incompatible avec la forme d'état proposée → STOP, décrire la
structure réelle et proposer un adaptateur, sans inventer de persistance.
