## Étape 1 — `InfoHover` : survol + clic verrouillant (décision Thibault)

Comportement cible (le composant reste **générique**, aucun contenu en dur) :

1. **Survol / focus** : ouvre le panneau, comme aujourd'hui ; le quitter le ferme — sauf si verrouillé.
2. **Clic (ou Enter/Espace) sur le déclencheur** : bascule un état `locked`. Verrouillé, le panneau
   reste ouvert quand la souris sort ; re-clic (ou Escape) le ferme et déverrouille.
3. **Clic hors du composant** pendant le verrouillage : ferme (listener `pointerdown` sur
   `document`, ajouté seulement quand `locked`, nettoyé au démontage — expliquer en commentaire).
4. Accessibilité : le déclencheur garde `role="button"` et gagne `aria-expanded={open}` ;
   Escape ferme même verrouillé. Le panneau garde `role="tooltip"`.
5. Le `onClick` **ne doit pas remonter** (`stopPropagation`) : les déclencheurs vivront dans des
   vignettes elles-mêmes cliquables (S3) — le clic info ne doit jamais ajouter un aliment.

Nouvelle prop optionnelle éventuelle : aucune — ce comportement devient LE comportement du
composant (il n'a aucun consommateur à préserver). Mettre à jour le commentaire d'en-tête
(« survol + clic verrouillant », mention de la décision 2026-07-09).

CSS : prévoir une largeur max de panneau plus généreuse (~280 px) et un `z-index` suffisant pour
sortir des cartes ; le panneau doit rester lisible à ~1 m (texte ≥ 14 px).

## Étape 2 — `CourbeGlycemie` : extensions 100 % optionnelles

Quatre ajouts, chacun inerte si non utilisé :

### 2.a Variantes duo (identité à poids égal)

`CourbeVariante` gagne `'duoA' | 'duoB'`. Même épaisseur/opacité que `principale` ; couleurs dans
`CourbeGlycemie.module.css` : `duoA` = `var(--color-nav)`, `duoB` = prune locale
(`oklch(50% 0.11 330)` — PAS un token global, PAS confort/vigilance/toxique). Exporter aussi les
classes ou une constante `DUO_COLOR` utilisable par S3 pour teinter les cartes assorties
(ex. `export const DUO_CSS_COLOR: Record<'duoA' | 'duoB', string>` contenant la valeur CSS).

### 2.b Marqueur de pic

```ts
/** Sommet de la courbe, coordonnées LOCALES du graphe (même repère que `d`). */
picAt?: { x: number; y: number };
```

Si présent : un point (r≈5, couleur de la courbe, halo surface) au sommet. Le composant reste
bête : c'est le module qui calcule le pic (il a les points échantillonnés).

### 2.c Étiquette directe

```ts
/** Étiquette nominative posée près du pic (nécessite picAt). */
etiquette?: string;
```

Si présent (avec `picAt`) : le `label` court rendu en `<text>` au-dessus/à côté du point de pic,
couleur de la courbe, taille ≥ 13 px dans le viewBox. **Anti-collision minimal** : si deux
étiquettes ont des ancres à moins de ~24 px verticaux et ~120 px horizontaux, décaler la seconde
au-dessous du point au lieu d'au-dessus (règle simple, pas de moteur de layout). Garder l'étiquette
dans le viewBox (clamp x).

### 2.d Tracé animé

Prop composant `animerTrace?: boolean`. Si vrai : chaque `path` de courbe reçoit `pathLength={1}`
et une classe CSS qui anime `stroke-dashoffset` 1→0 (~900 ms, ease-out) **au montage**. Les
consommateurs déclenchent l'animation en montant le composant au moment voulu (S3 : au révèle —
le composant n'apparaît qu'à ce moment-là, aucun besoin de re-trigger). Respecter
`@media (prefers-reduced-motion: reduce)` : pas d'animation. Le marqueur de pic et l'étiquette
apparaissent en fondu **après** le tracé (delay CSS ≈ 900 ms).

### Non-régression (bloquant)

- Aucun changement de signature obligatoire ; `variante` existantes intactes ; aucun appel existant
  modifié. Après build : relire les usages (`grep CourbeGlycemie src/features/diabete`) et
  confirmer dans le rapport qu'aucun ne passe les nouvelles props.
