## E7 — Carnet de suivi des consommations (localStorage)

### Objectif
Un carnet permettant au patient de consigner ses consommations (saisie rapide quotidienne), avec
historique simple, persisté strictement en localStorage sur son appareil.

### Décision clé
Outil personnel du patient (décision D8 : localStorage autorisé côté patient). Réutiliser l'util
`storage.ts` (S12). Confidentialité : afficher clairement que les données restent sur l'appareil +
bouton d'effacement ; gérer le cas « stockage vidé par le navigateur ».

### Lire / Modifier
- Créer : `src/patient/carnet/PatientCarnet.tsx` + `.module.css`
- Modifier : navigation/accueil patient (ajouter l'entrée d'accès au carnet)

### Étapes
1. Format de stockage : clé `etp.patient.carnetConso`, JSON `[{ id, dateHeure, contexte, ressenti }]`
   (une entrée = une consommation). Lecture/écriture via `storage.ts`.
2. UI : formulaire de saisie rapide — `date et heure` pré-remplie à **maintenant** (modifiable),
   `contexte`, `ressenti` ; liste chronologique des entrées ; édition + suppression d'une entrée.
   Courbe simple = option ultérieure (v1 = liste).
3. Confidentialité : bandeau « Ces données restent sur votre appareil » + bouton « Effacer le carnet »
   (avec confirmation). Gérer localStorage indisponible (repli mémoire, pas de crash).
4. Accès : entrée depuis l'accueil patient (carte/bouton), cohérente avec la navigation existante.

### Validation
- Auto (bloque le commit) : `npx tsc --noEmit` + `npm run build` (2 entrées) + `npm test` → OK.
- Humain : ajouter/éditer/supprimer une entrée ; recharger → données conservées ; « Effacer » vide le
  carnet ; message de confidentialité présent → `VALIDATION.md`.

### Si bloqué
Ambiguïté sur les champs (au-delà de la proposition) → livrer la v1 proposée, marquer les extensions à
revalider, signaler.
