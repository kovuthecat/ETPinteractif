# S1 — Fondation : chrome diabète (onglets dans le header + contenu élargi)

> **Mode** : solo · **Exécutant** : Sonnet · **Effort** : xhigh · **Dépend de** : — (à faire **en premier**).
> **À lire avant** : index v3 §2 · `src/components/ModuleShell.tsx` + `.module.css` · un module diabète avec onglets pour voir le pattern (`src/features/diabete/risque-cardio/RisqueCardioModule.tsx`, barre `VUES` l.174-187).

## Problème (audit 1.2 + 2)

`ModuleShell` impose à **tous** les modules (tabac + diabète) :
- `.header` et `.content` plafonnés à `max-width: 980px` → sur un écran de 1534 px, ~277 px de marge morte de chaque côté et un contenu utile de **~900 px** (`980 − 2×40` de padding).
- Le **titre** (`<h1>`) vit dans `.header` ; la **barre d'onglets** de chaque module est rendue **dans les `children`** (donc sous le titre, sur sa propre ligne).

Thibault veut, **côté diabète uniquement** : la barre d'onglets **sur la même ligne que le titre**, et un contenu **plus large** (moins de marge morte, objets pédagogiques plus grands). Le `ModuleShell` étant partagé et le moteur devant rester **thème-agnostique** (invariant 4), on procède par **props optionnelles** ; le tabac ne les passe pas et reste identique au pixel.

## Résultat attendu

1. `ModuleShell` accepte deux props optionnelles **agnostiques** :
   - `nav?: ReactNode` — rendu dans le `.header`, **à droite du titre, sur la même ligne**, repli sous le titre quand la place manque.
   - un contrôle de largeur, ex. `wide?: boolean` (ou `contentMaxWidth?: number`) — élargit `.header` **et** `.content` (cible de départ **~1240 px**) et resserre les marges.
2. Les **9 modules diabète** passent `wide` et, pour ceux qui ont une barre d'onglets, la remontent dans `nav`.
3. **Aucun changement de rendu pour le tabac** (props absentes → comportement actuel).
4. Le module Mécanisme (`.scene width:100%`) devient automatiquement plus grand grâce à l'élargissement — vérifier qu'il n'y a pas de régression de positionnement.

## Travail

### 1. `ModuleShell.tsx` / `.module.css` — les deux props

- **`nav` slot** : dans `ModuleShell.tsx`, après le `<h1>`, rendre `{nav && <div className={styles.nav}>{nav}</div>}`. Passer `.header` en `flex-wrap: wrap` avec le titre en `flex: 1 1 auto` et `.nav` en `flex: 0 1 auto` → même ligne si ça tient, sinon la nav passe sous le titre (largeur pleine). Conserver `.back` et `Sources` à leurs extrémités.
  - Attention à l'ordre visuel : aujourd'hui `.header` est `back | titre | sources`. Cible : `back | titre + nav | sources`, ou `back | titre | sources` sur la 1ʳᵉ ligne puis `nav` en 2ᵉ ligne si trop large. Choisir la structure flex la plus simple qui donne « nav à droite du titre, repli propre ». Documenter le choix.
- **Largeur** : introduire `wide`. En CSS, une classe `.contentWide` / `.headerWide` (ou une variable `--shell-max`) portant `max-width: 1240px`. Appliquer conditionnellement aux deux conteneurs. Garder le `padding-inline: 40px` (ou le réduire légèrement) — le gain vient surtout du `max-width`.
- **Ne rien changer** au chemin par défaut (props absentes) : `max-width: 980px`, pas de `.nav`.

### 2. Câbler les 9 modules diabète

Pour **chacun** des modules sous `src/features/diabete/*/` :
- Ajouter `wide` (ou `contentMaxWidth={1240}`) au `<ModuleShell>`.
- **Si le module a une barre d'onglets de premier niveau** (boutons de bascule de vue), la **déplacer** de `children` vers la prop `nav`. Le JSX de la barre **reste dans le module** (spécifique) — on ne fait que le passer par le slot. Modules concernés (barre à remonter) : **risque-cardio** (`VUES`), **alimentation** (défis), **activité** (onglets), **suivi**, **hypoglycémie**, **insuline** (temps ①②③). Modules **sans** barre de premier niveau : **mécanisme** (boutons de mode = contenu, ne pas remonter), **complications**, **traitements** (bascule ligne/tous = contrôle interne, laisser dans le contenu). Trancher au cas par cas en lisant chaque module ; en cas de doute, **laisser la barre dans le contenu** et se contenter de `wide` (le gain de largeur reste acquis).
- Vérifier que la barre déplacée conserve son état actif et son style (les classes `.navigation`/équivalentes du module peuvent avoir besoin d'un léger ajustement pour vivre dans le header — pas de largeur pleine forcée).

### 3. Mécanisme — profiter de l'élargissement

- Avec `wide`, `.scene` (`width:100%`, `aspect-ratio: 1060/340`) s'agrandit toute seule (~1160 px de large au lieu de ~900). Vérifier visuellement que les éléments positionnés en `%` (pancréas, cellules, clés) restent bien calés et que `min-height` n'écrête rien.
- Si Thibault veut aller **au-delà** de la largeur de contenu (full-bleed), c'est un réglage ultérieur — ne pas l'implémenter ici sans validation (risque de scroll horizontal). Le noter dans `VALIDATION.md`.

### Écart par rapport au plan
Le plan supposait implicitement que les modules diabète appelaient déjà `<ModuleShell>`
directement ; en réalité seul `App.tsx` le fait (wrapping générique). Pour remonter la barre
d'onglets dans le slot `nav` (qui vit dans le header, rendu par un ancêtre), il a fallu inverser
le contrôle : `ModuleDef.rendersOwnShell` (générique, pas de nom de thème) fait qu'`App.tsx`
délègue le rendu de `ModuleShell` au module lui-même, qui reçoit `titre`/`sources`/`onBack` via
`ModuleProps.shell`. Le tabac ne passe jamais ce flag → chemin `App.tsx` historique inchangé.
Mécanisme géométrique (Bézier plaque), feux 3/2, alimentation, etc. non traités ici (S3/S4/…).
