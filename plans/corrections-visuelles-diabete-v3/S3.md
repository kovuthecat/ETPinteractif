## 3.1 — Les 5 feux en 4/1 → cible 3/2

**Cause** : `.feuxRow` (`RisqueCardioModule.module.css` ~l.52-57) est `display:flex; flex-wrap:wrap` **sans `max-width`**. Cartes `.feuCard` = 200 px, gap `--space-md` 16. Sur ~900 px (et pire après S1, ~1240 px), 4 (voire 5) cartes tiennent sur la 1ʳᵉ ligne → 4/1 (ou 5/0).

**Correctif** : borner `.feuxRow` pour n'autoriser que 3 cartes par ligne :
```css
.feuxRow { max-width: calc(200px * 3 + var(--space-md) * 2); } /* = 632px, centré par .vueBody */
```
Toute valeur dans `[632, 847]` px force le 3/2 sur la largeur d'une carte à 200 px. `justify-content:center` (déjà là) centre les 2 cartes de la 2ᵉ ligne.
⚠️ **Après S1**, si les cartes sont agrandies, recalculer la borne (`n×largeurCarte + (n−1)×gap`). Valeur `// à caler`.

## 3.2 — La plaque ne réduit jamais la lumière (BUG Bézier)

**La chaîne état→% est correcte** : `scoreSum`/`score` (`RisqueCardioModule.tsx` ~l.148-149), passé en `<PlaqueArtere encrassement={score} />` (~l.260) ; `plaquePassagePct(1) = 30 %` (`PlaqueArtere.tsx` ~l.29-33). Le texte « Passage du sang : 30 % » est donc juste. **C'est la traduction en géométrie SVG qui est fausse.**

Le dessin (`PlaqueArtere.tsx` ~l.38-40) :
```js
function crescentPath(edgeY, peakY) {
  return `M0,${edgeY} Q50,${peakY} 100,${edgeY} Z`;
}
```
`peakY` est injecté comme **point de contrôle** d'une Bézier quadratique, mais le code le suppose être l'**apex**. Or pour `Q P0(0,edgeY) P1(50,peakY) P2(100,edgeY)`, le point réel au milieu (t=0.5) vaut :
```
y(0.5) = 0.25·edgeY + 0.5·peakY + 0.25·edgeY = (edgeY + peakY) / 2
```
→ **la plaque n'atteint que la moitié** de la profondeur visée. À `e=1`, `wallDepth=35` : l'apex réel n'est qu'à **17,5** (au lieu de 35) côté haut, et à **82,5** (au lieu de 65) côté bas → lumière visuelle ≈ **65 %** alors que le texte annonce 30 %. Les deux croissants restent **collés aux parois** = exactement le symptôme de l'audit.
**Ce que v2 a fait** : `oppositeDepthFactor = 0.5 + 0.5·e` (~l.51-53) n'a corrigé que la **symétrie** des deux dépôts, jamais la division par deux. D'où « v2 inopérante ».

**Correctif** — compenser le facteur ½ pour que l'apex réel atteigne la profondeur voulue. Réécrire `crescentPath` en prenant l'apex désiré et en doublant en interne :
```js
function crescentPath(edgeY, apexY) {
  const ctrl = 2 * apexY - edgeY;            // apex = (edgeY + ctrl)/2  ⇒  ctrl = 2*apex − edgeY
  return `M0,${edgeY} Q50,${ctrl} 100,${edgeY} Z`;
}
```
Les appels (~l.68 haut, ~l.71 bas) restent sémantiquement « apex » : `crescentPath(0, wallDepth)` et `crescentPath(100, 100 - wallDepth * oppositeDepthFactor(e))`. À `e=1` : apex 35 / 65 → lumière = 30 unités = **30 % visuel = 30 % annoncé**, plaque ≈ 70 % du diamètre.
*(Variante minimale sans toucher la fonction : doubler dans les appels — `crescentPath(0, 2*wallDepth)` et `crescentPath(100, 100 - 2*wallDepth*oppositeDepthFactor(e))`. Préférer la réécriture, plus lisible.)*

**À vérifier après le fix (validation humaine)** : le calage de la boîte `.artereOverlay` (`RisqueCardioModule.module.css` ~l.181-189, `62%×30%`, `left/top`, `rotate(-25deg)`) sur la lumière réelle de `artere-saine.png` — un léger décalage deviendra plus visible une fois la plaque profonde. Ne **pas** régénérer l'image ; ajuster seulement le positionnement de l'overlay si besoin.

**Ne pas** modifier `plaquePassagePct` (source de vérité du texte et de l'`aria-label`). On aligne la géométrie sur elle. Garder `BILATERAL_THRESHOLD = 0.5` (dépôt unilatéral sous 3 facteurs) et la `transition: d 0.6s` de `.plaque`.

## 3.3 — Anatomie : pin découplé des feux + texte de résultat mis en valeur

**Couplage (bug)** : le pin (`plaque.png`) est rendu sous garde `rougeCount > 0 && zoneActive && (…)` (`RisqueCardioModule.tsx` ~l.279), avec `rougeCount` dérivé de l'état **partagé** `factors` (modifié dans l'onglet ① Leviers). Tant qu'aucun feu n'est rouge, cliquer une zone n'affiche jamais de pin.
**Correctif** : retirer la seule condition `rougeCount > 0 &&`, garder `zoneActive && (…)`. Le pin est purement illustratif (aucun paramètre de sévérité), donc le montrer toujours est cohérent. Ne pas retirer la garde `zoneActive` (sinon pin sans sélection). `rougeCount` sert encore à `arteryMessage` (~l.158) → **le laisser défini**, juste ne plus s'en servir comme garde du pin.

**Texte de résultat** (`.zoneDescPanel`, ex. « c'est l'infarctus ») : aujourd'hui la colonne `.zonesCol` (max-width 420) + silhouette `.silhouetteWrap` (560) = 1020 > 900 → `flex-wrap` fait **retomber** le texte sous la silhouette, et le texte est à la taille de base (trop petit).
**Correctif** (double, à recalibrer avec la largeur S1) :
1. **À côté** : après S1 (contenu ~1240 px), `560 + gap + colonne` tient sur une ligne. S'assurer que `.anatomieRow` garde le texte à droite (réduire le gap à `--space-lg` si besoin, borner `.zonesCol`, `flex-wrap:nowrap` au-dessus d'un breakpoint et `wrap` en dessous pour le mobile).
2. **Plus grand** : ajouter une `font-size` sur `.zoneDescPanel p` (~l.238), ex. `clamp(1.15rem, 2.4vw, 1.5rem)` (lisible à ~1 m).
Le pin (`.plaquePin`, positionné en % dans le `.wrap` de la silhouette) **suit** la silhouette — réduire la colonne texte ne le déplace pas.
