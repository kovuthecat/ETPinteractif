### 0.a Assiette vide → courbe plate (pour B1)

`peakAmplitude` impose un plancher `AMP_FLOOR_FRACTION` (6 %) même quand `charge === 0` :
une assiette vide dessine une petite bosse. Corriger : **si `charge === 0`, amplitude = 0**
(le plancher ne s'applique qu'à partir du premier aliment). Test : `paramsFromAssiette({aliments: []})`
→ `sampleRepas` reste exactement à `BASELINE` partout.

### 0.b Effet d'ordre gradué (pour B2)

Remplacer le booléen `ordreFeculentDernier` de `Assiette` par un champ gradué :

```ts
/** Position du féculent dans le repas, 0 = mangé en premier … 1 = mangé en dernier. */
ordreFeculent?: number; // clampé [0,1]
```

Dans `paramsFromAssiette` : `ordreBonusFrein = ORDRE_FREIN_BONUS * clamp01(ordreFeculent)`,
idem pour `ORDRE_RETARD_BONUS`. **Migration des appelants** (le booléen disparaît, pas de compat) :

- `AlimentationModule.tsx` défi 3 : `ordreFeculent = d3FeculentIdx / (d3Foods.length - 1)`.
- Synthèse ★ : `ordreFeculentDernier: true` → `ordreFeculent: 1` ; version naïve `false` → `0`.
- Tests existants sur l'ordre : adapter, puis **ajouter l'invariant** :
  pic(féculent en premier) > pic(féculent au milieu) > pic(féculent en dernier), à assiette égale.

### 0.c Refonte du modèle de repas : composition nutritionnelle réelle (pour B1/B3 — demande Thibault du 2026-07-09)

**Décision** (remplace le mécanisme actuel) : la courbe ne se calcule plus par heuristiques
de familles ni par proximité à l'assiette-modèle — elle dérive de la **composition
approximative des aliments** : charge glycémique, fibres, protéines, lipides. Fonctionnement
proche de la réalité physiologique, même si approximatif (ordres de grandeur, pas un
simulateur — l'avertissement d'en-tête de la lib reste vrai).

1. **Données réelles** (`alimentation/data.ts`) — décision Thibault : les modules **gardent**
   la notion CG basse / moyenne / haute (pastilles vert/orange/rouge, libellés inchangés),
   mais la table qui alimente les courbes passe aux **vraies valeurs par portion usuelle** :
   - `cg` n'est plus l'échelle relative 0-100 : c'est la **charge glycémique réelle de la
     portion** (IG × glucides de la portion / 100, ordres de grandeur des tables
     internationales GI/GL). Réécrire l'en-tête de `data.ts` en conséquence.
   - Seuils des pastilles = **seuils standards publiés** : `cgTier` devient
     basse ≤ 10 · moyenne 11-19 · haute ≥ 20 (remplace 25/65). Quelques pastilles bougent
     par rapport à la maquette (pita et galette de riz passent orange, couscous passe rouge…)
     — la réalité prime, tout est `// à revalider (Thibault)`.
   - Nouveaux champs `fibres` / `proteines` / `lipides` : **grammes réels par portion**.

   Table de départ (approximations honnêtes, ajustables si un invariant l'exige en restant
   dans les ordres de grandeur réels) :

   Le champ `portion` peut rester un simple commentaire par ligne (pas besoin d'un champ
   TypeScript) — il documente à quoi correspondent les valeurs pour la revalidation.

2. **Types lib** — `AlimentRepas` devient `{ cg: number; fibres: number; proteines: number;
   lipides: number }` (la `famille` sort de la lib : `FAMILLES_FREIN` et le champ
   `proportions` de `Assiette` **disparaissent** ; l'ordre du féculent reste le champ gradué
   `ordreFeculent` de 0.b, calculé par les modules qui, eux, connaissent les familles).
   `RepasParams { charge, frein, retard }` et les fonctions de forme (`peakAmplitude`,
   `peakTimeMinutes`, `endTimeMinutes`) **ne changent pas d'interface** — seul
   `paramsFromAssiette` change de dérivation. Modules 3 et 9 (`REPAS_REFERENCE`,
   `REPAS_JOURNEE`) intouchés.

3. **Dérivation** (`paramsFromAssiette`) — physiologie qualitative :
   - `charge` = saturation de ΣCG (inchangé : `1 − exp(−ΣCG / K_CHARGE)`).
   - **fibres** : freinent fort l'absorption (aplatissent), retardent modérément le pic ;
   - **lipides** : ralentissent fort la vidange gastrique (retardent le pic, allongent la
     descente via `frein`→`endTime`), aplatissent modérément ;
   - **protéines** : aplatissent modérément, retardent peu.
   Structure suggérée (poids de départ, à calibrer par les tests) :
   `frein = 1 − exp(−(1.0·ΣF + 0.25·ΣL + 0.15·ΣP) / K_FREIN)` et
   `retard = 1 − exp(−(0.5·ΣF + 1.0·ΣL + 0.15·ΣP) / K_RETARD)`, plus la contribution
   `ordreFeculent` de 0.b sur les deux. Choisir `K_FREIN`/`K_RETARD` pour que les assiettes
   types du module restent dans la dynamique visuelle actuelle (pic entre ~40 et ~95).

4. **Invariants testés** (chacun = une affirmation pédagogique du brief, tous sur assiettes
   à portions comparables) :
   - baguette vs pain complet, riz blanc vs basmati vs complet : version complète → pic
     plus bas **et** plus tardif ;
   - riz blanc + brocoli < riz blanc seul (fibres) ; riz blanc + huile d'olive : pic plus
     bas et nettement plus tardif (lipides) ; riz blanc + poulet : pic plus bas (protéines) ;
   - lentilles ≪ riz blanc (légumineuses : CG basse + fibres — la « pépite » du brief) ;
   - galette de riz : pic haut et précoce malgré l'image « léger » (fibres quasi nulles) ;
   - pastèque : petit pic (CG basse malgré l'IG réputé haut — piège du brief §6.5) ;
   - **B3 émergent** : à portions totales constantes, remplacer une portion de féculent par
     une portion de protéine ne fait jamais monter le pic ;
   - **assiette-modèle émergente** : ½ légumes · ¼ protéines · ¼ féculents plus douce que
     100 % féculents à portions égales (sans aucun paramètre « proportions » dans la lib).

5. **Recalibrer le défi 2** : les seuils `PEAK_BAS_MAX`/`PEAK_HAUT_MIN` de `classifyPeak`
   (`AlimentationModule.tsx:61-62`) ont été calibrés sur l'ancien modèle — ré-échantillonner
   le garde-manger complet avec le nouveau et reposer les deux seuils pour que les
   comparaisons intra-famille du brief (baguette/pain complet, riz blanc/basmati/complet)
   tombent dans des classes différentes.

## B1 — Composition (défi 1 M2) : « je n'arrive pas à glisser d'autre aliment dans l'assiette ; je dois pouvoir glisser plusieurs féculents ; assiette vide de base ; pas de cercle au centre »

**Fichiers** : `alimentation/AlimentationModule.tsx` + `.module.css`.

**Causes racines** (`AlimentationModule.tsx:192-222`) : le défi 1 impose un féculent de base
(`d1Base = 'riz-blanc'`, cercle central `styles.d1Base`), un seul féculent (tout drop de
féculent *remplace* la base), ignore silencieusement fruits/laitiers (`applyD1Food` return),
refuse les doublons et plafonne à 3 ajouts → la plupart des glisser-déposer ne font rien.

**Correctif** — le défi 1 devient une assiette libre (même patron que la synthèse ★) :

- État : `d1Plate: { uid: string; id: string }[]`, **vide au montage**. Supprimer
  `d1Base`/`d1Additions`/`D1_ADDITION_ANCHORS` et le cercle central `d1Base` (l'assiette reste
  le grand cercle pointillé, les aliments s'y posent en chips cliquables — clic = retirer).
- Accepter **toutes les familles** (fruits et laitiers compris), **les doublons** (plusieurs
  portions du même féculent), plafond ~10 comme la synthèse.
- Courbe : `buildCourbe({ aliments: d1Foods.map(toAliment) })` — grâce à 0.a, assiette vide
  = trait plat sur la baseline (on garde la carte courbe affichée).
- Captions « un ajout à la fois » conservées mais pilotées par le contenu réel :
  assiette vide → « Glissez un premier aliment depuis le garde-manger. » ;
  que des féculents → caption « grand pic » ; féculents + 1/2/≥3 aliments-frein
  (protéine/lipide/légume) → captions existantes `D1_CAPTIONS[1..3]`. La notion
  « aliment-frein » ne vit plus que côté module (par famille, pour les captions et les
  principes ✓ de la synthèse) — la courbe, elle, réagit à la composition réelle (0.c) :
  fruits et laitiers y contribuent par leur CG/fibres/protéines/lipides.
- Panneau latéral : remplacer « Féculent : X » / « Ajouts (n/3) » par un compte simple
  (« Dans l'assiette : n aliments, dont k féculents ») + « Recommencer ».
- `handleShelfActivate(defi 1)` → ajoute à l'assiette (même geste que le drop).

**Acceptation** : au chargement, assiette vide sans cercle central, courbe plate ; on peut
déposer 2 riz + 1 pain + 1 fruit + 1 laitier ; chaque dépôt modifie la courbe ; clic sur un
aliment posé le retire ; « Recommencer » vide tout.

## B3 — Proportion (défi 4 M2) : « la courbe augmente quand j'augmente les protéines au détriment des féculents »

**Fichier** : `alimentation/AlimentationModule.tsx` (+ invariant lib 0.c).

**Cause racine** (`AlimentationModule.tsx:330-334`) : `d4Aliments` = un seul aliment
représentatif par catégorie non nulle → la `charge` est **constante** quelles que soient les
portions ; seule joue la proximité à l'assiette-modèle ½·¼·¼, symétrique — s'éloigner du
modèle en *réduisant* les féculents fait perdre le bonus de frein et **monte** la courbe.

**Correctif** (le paramètre `proportions` n'existe plus, cf. 0.c) : construire l'assiette
avec les portions réelles — chaque catégorie apporte son aliment représentatif **répété
`d4Counts[cat]` fois** :

```ts
const d4Aliments = D4_CATEGORIES.flatMap((cat) => {
  const food = foodById(cat.repFoodId);
  return food ? Array.from({ length: d4Counts[cat.id] }, () => toAliment(food)) : [];
});
const d4Courbe = buildCourbe({ aliments: d4Aliments });
```

L'effet des proportions **émerge** de la composition : plus de riz = plus de CG (pic plus
haut), plus de brocoli/poulet = plus de fibres/protéines (pic plus bas et plus doux). Le
camembert et le message « ✓ proche de l'assiette-modèle » (comparaison ½·¼·¼) restent —
c'est de l'affichage, plus un paramètre de courbe.

**Acceptation** : depuis l'état initial (1 légume / 0 protéine / 3 féculents), chaque échange
« +1 protéine / −1 féculent » abaisse le pic ; l'assiette-modèle ½·¼·¼ donne une courbe
nettement plus douce que 100 % féculents ; tout à zéro → courbe plate.
